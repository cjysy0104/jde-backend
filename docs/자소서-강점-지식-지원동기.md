[버전 1 – 데이터·API 통합]
저의 강점은 외부 API 기반 데이터 수집·가공 작업을 안정적으로 수행하고, 이를 서비스 흐름에 맞게 통합하는 능력입니다.

봉사활동 커뮤니티 앱 개발에서는 1365 데이터 API를 활용해 하루 약 100건의 봉사 데이터를 수집했습니다. 데이터 양이 증가하면서 처리 지연이 발생해, API 응답 구조에 맞춘 페이징 처리를 적용해 불필요한 로직을 줄였고, 그 결과 응답 속도를 체감적으로 개선할 수 있었습니다.

전기차 커뮤니티 웹 개발에서는 충전소 정보 API와 KakaoMap API를 연동해 충전소 위치 기반 추천 기능을 구현했습니다. 주소 변환 API를 사용해 좌표 오차를 줄였고, MyBatis/JDBC 기반의 단순 캐싱 로직을 적용해 반복 조회되는 데이터의 조회 시간을 단축했습니다. 이를 통해 API 호출 비용과 서버 부하를 줄이는 효과를 얻었습니다.

맛집 리뷰 커뮤니티 서비스(JustDoEat) 개발에서는 레스토랑 목록·검색 API와 신고 기능을 담당했습니다. 레스토랑 목록 조회 시 각 행마다 대표 썸네일을 따로 조회하다 보니 N+1 쿼리로 20건 조회 시 21번의 DB 요청이 발생했습니다. MyBatis 스칼라 서브쿼리로 한 번의 SELECT에 썸네일을 포함하도록 수정해 쿼리 수를 1번으로 줄였고, 커서 기반 무한 스크롤(cursor, size)을 적용해 API 안정성을 높였습니다. 또한 댓글·리뷰 신고 등록(사용자)부터 관리자용 신고 목록·상세·키워드 검색·처리까지 신고 관련 기능 전부를 구현하며, 사용자와 관리자 흐름을 하나의 서비스로 통합하는 경험을 쌓았습니다.

이러한 경험을 통해 다양한 외부 데이터를 서비스에 맞게 통합하고, 데이터 처리 흐름과 DB 조회 성능을 개선하는 실무적인 역량을 갖추게 되었습니다.

[버전 2 – 구현 가속도]
저의 가장 큰 강점은 비즈니스 로직을 신속하게 API로 설계하고, 이를 프론트엔드 화면에 즉각적으로 통합하여 서비스 구현 속도를 높이는 '구현 가속도'입니다.

봉사활동 커뮤니티 앱에서는 1365 데이터 API 연동 요구사항을 받고, 페이징·응답 구조를 정리한 뒤 REST API를 설계하고 프론트에 연동해 봉사 목록 화면을 빠르게 구현했습니다. 전기차 커뮤니티 웹에서는 충전소 API와 KakaoMap을 연동해 지도 기반 추천 기능을, 주소·캐싱 로직을 API에 반영해 화면에서 바로 사용할 수 있도록 설계했습니다. 맛집 리뷰 서비스(JustDoEat)에서는 레스토랑 목록·검색·무한 스크롤과 댓글·리뷰 신고(사용자·관리자)까지 요구사항을 API 스펙으로 정리하고, 커서 기반 페이지네이션을 적용한 뒤 N+1 해결로 조회 성능을 맞춰 목록·검색·신고 화면이 안정적으로 동작하도록 했습니다. 비즈니스 요구를 API로 빠르게 옮기고, 그 API를 화면과 바로 연결해 전체 구현 속도를 높이는 방식을 지속적으로 적용해 왔습니다.
